OpenSeek 超长上下文数据标注技术方案

一、方案概述

1.1 背景与目标
本方案面向 OpenSeek 赛题，目标是在超长上下文条件下，通过可复现、可扩展的提示工程与多轮交互机制，稳定完成 8 个任务的数据标注与代码生成。方案同时满足两类评审要求：
1. 预测结果成绩（60%）：提升任务 1-7 的标注准确率与任务 8 的代码可执行性、功能正确性。
2. 技术方案成绩（40%）：围绕超长上下文指令设计、超量示例压缩组织、多轮交互一致性给出可落地工程解法。

1.2 核心思路
我们采用“任务解耦 + 指令模板化 + 结果解析标准化 + 两阶段复核”的统一框架：
1. 任务解耦：每个任务一个独立脚本，按任务类型定制 prompt 和后处理。
2. 指令模板化：所有任务均由 Task Definition/Rules/Examples/Target 组成，减少指令歧义。
3. 结果解析标准化：针对不同任务设计 count_answer 规则，尽量从 </think> 后提取最终答案，避免中间推理噪声干扰。
4. 两阶段复核：任务 4/7/8 在首轮生成后支持二次审核与即时回写，提升最终一致性和稳定性。

1.3 系统特点
1. 统一本地推理接口：OpenAI 兼容本地服务（http://localhost:2026/v1）。
2. 全流程断点续跑：按 test_sample_id 跳过已完成样本。
3. 实时落盘：逐条写入 jsonl，降低长任务中断损失。
4. 顺序重写：复核阶段保证输出按原样本顺序重排写回，便于评测与追踪。


二、整体架构与执行流程

2.1 代码组织
1. 任务脚本：src/1.py 到 src/8.py。
2. 统一调度：src/assign.py（交互式选择任务并顺序执行）。
3. 对比基线：src/run.py（旧统一方案入口）。
4. 输入数据：data/openseek-*.json。
5. 输出结果：outputs/result/openseek-*-v1.jsonl。

2.2 统一处理链路
步骤 A：读取数据集 Definition、examples、test_samples。
步骤 B：按任务构造 ICL Prompt（含规则和示例）。
步骤 C：调用本地模型生成结果。
步骤 D：按任务解析答案并标准化。
步骤 E：写入 jsonl（逐条 flush）。
步骤 F：若支持复核（任务 4/7/8），在第二次运行进入审核流程并即时回写。

2.3 结果格式
统一输出 jsonl，每行结构：
{"test_sample_id": "...", "prediction": "..."}


三、面向赛题核心问题的技术设计

3.1 核心问题一：超长上下文下如何设计有效指令与提示策略

3.1.1 结构化指令骨架
我们将提示分解为四层：
1. Definition 层：绑定任务目标，减少任务漂移。
2. Rules 层：显式约束输出空间（如仅输出数字、Y/N、小写实体、代码块）。
3. Examples 层：通过样本演示固定输出风格。
4. Target 层：对当前样本进行逐条求解。

3.1.2 任务特定约束
1. 任务 1/2/3：数字或列表任务，规则明确、解析严格。
2. 任务 4：字符串拼接任务，强调字符级一致性（不增删改）。
3. 任务 5：情感二分类任务，强调标签边界与歧义处理。
4. 任务 6：二值判断任务，输出限定为 Y/N。
5. 任务 7：问答任务，输出强制小写实体。
6. 任务 8：代码生成任务，输出限定 python 代码块并进行接口一致性检查。

3.1.3 推理噪声抑制
考虑到模型可能输出长推理过程，方案优先解析 </think> 后的文本作为最终答案区域，降低推理链文本对抽取结果的干扰。


3.2 核心问题二：示例数量远超上下文容量时，如何构造信息密集且结构合理的超长输入

3.2.1 分层示例选择
1. 常规任务（1-7）：采用少样本抽样或末端高质量示例拼接，保持输入紧凑。
2. 任务 5：按类别分组抽样（Sad/Not sad），保证类间平衡，减少偏置。
3. 任务 7：随机抽取固定 shot 数，增强覆盖面与鲁棒性。

3.2.2 长代码任务（任务 8）专用策略
1. 候选池随机采样：先从全量 examples 中随机抽取固定规模候选池，降低全量检索带来的上下文开销。
2. embedding 重排：对当前样本 query_text 与候选池示例进行向量相似度匹配，选出相似度最高的 top-5 参考样例。
3. 长度门控与截断：限制参考代码长度，避免超过上下文预算，并对过长参考代码做截断。
4. 输入结构解析：将原始输入拆分为 Functional Description、Wrapper Entry Information、Math 等字段，提升信息对齐度。
5. 接口签名对齐：从 Wrapper 信息中提取关键签名并注入 prompt，提升生成代码可调用性。

3.2.3 信息密度提升手段
1. 去冗余：去除无关说明与重复语句。
2. 强模板：通过固定步骤描述生成框架，降低模型输出发散。
3. 输出约束：通过格式规则将 token 用于有效信息而非冗余解释。


3.3 核心问题三：自动多轮对话/持续交互中如何兼顾一致性与可扩展性

3.3.1 两阶段执行机制
第一阶段：批量生成并保存首轮答案。
第二阶段：复核模式（人工确认后进入），逐题复审并修正。

3.3.2 复核机制（任务 4/7/8）
1. 触发条件：当检测到全部样本已生成，再次运行脚本时提示是否进入复核。
2. 复核提示：将输入与候选答案一并送入模型，要求仅保留或修正最终答案。
3. 即时回写：一旦修正发生，立即重写 jsonl，确保中断时不丢失改动。
4. 顺序一致：按原始样本顺序写回，保证评测稳定。

3.3.3 可扩展交互（任务 8）
任务 8 复核支持指定“从第几题开始”，适用于长时任务中断恢复和重点区间复查，显著提升持续交互效率。

3.3.4 稳定性增强
1. 缺失 </think> 自动重试（任务 4/7）。
2. 异常捕获与默认回退（prediction=no 或模板代码兜底）。
3. 结果字典缓存 + 增量写入，降低重复计算成本。


四、8个任务的实现映射

4.1 任务 1（Closest Integers）
1. 策略：步骤化比较整数差值。
2. 输出：单整数。
3. 解析：优先取 </think> 后最后整数。

4.2 任务 2（Count Nouns/Verbs）
1. 策略：显式词性计数规则。
2. 输出：整数。
3. 解析：正则抽取 + 多候选统计。

4.3 任务 3（Collatz）
1. 策略：严格按序列规则推导。
2. 输出：整数列表字符串。
3. 解析：校验列表格式，非法结果剔除。

4.4 任务 4（Conala Concat Strings）
1. 策略：字符级拼接约束。
2. 输出：无空格、无额外符号的拼接串。
3. 复核：二次校对字符一致性并即时回写。

4.5 任务 5（Tweet Sadness）
1. 策略：二分类规则 + 类别分组示例采样。
2. 输出：Sad/Not sad。
3. 解析：优先识别 final answer 模式，其次关键词回退。

4.6 任务 6（MNLI Same Genre）
1. 策略：强约束输出 Y/N。
2. 输出：Y 或 N。
3. 解析：从后向前选择高置信匹配。

4.7 任务 7（Jeopardy Answer）
1. 策略：线索推理 + 小写实体输出。
2. 输出：小写实体（可多答案逗号分隔）。
3. 复核：候选答案一致性审核与即时回写。

4.8 任务 8（Kernel Generation）
1. 策略：随机候选池采样 + embedding 相似度重排 + top-5 参考代码注入 + 输入字段拆解 + 结构化代码模板。
2. 输出：python 代码块。
3. 复核：功能描述与 wrapper 对齐检查；支持从指定题号继续复核。
4. 兜底：当抽取失败时使用最小模板代码，保证可提交性。


五、评测与复现方案

5.1 评测指标对齐
1. 任务 1-7：以准确率为主。
2. 任务 8：关注代码可运行率与计算正确率。

5.2 实验组织
1. 主方案：python assign.py（支持 all 或指定任务）。
2. 单任务：python 1.py ... python 8.py。
3. 复核任务：任务 4/7/8 首轮完成后再次运行进入 review。
4. 对比基线：python run.py --task_id 8 --backend nvidia --log_path_prefix ../outputs/result/

5.3 复现关键条件
1. 统一推理端点：本地 OpenAI 兼容服务。
2. 统一模型路径：/root/ascend/log/debug/Qwen3-4B。
3. 固定输出路径与命名规范：openseek-<task>-v1.jsonl。
4. 提供 requirements.txt 与运行说明，保证可复现。

七、工程可用性与风险控制

7.1 工程可用性
1. 支持断点续跑、增量写入、交互式任务调度。
2. 支持二次运行复核，适配长任务与多人协作复查场景。

7.2 风险与应对
1. 风险：模型输出格式漂移。应对：正则多路解析 + 默认值兜底。
2. 风险：长任务中断导致结果丢失。应对：逐条 flush + 复核即时回写。
3. 风险：超长提示超预算。应对：示例筛选、截断与结构化压缩。


八、与赛题提交要求的对应关系

8.1 对“超长上下文提示策略”的响应
通过 Definition/Rules/Examples/Target 的结构化提示模板，结合任务特定规则，实现稳定输出。

8.2 对“示例显著超过容量”的响应
通过分层抽样、随机候选池 + embedding 重排、长度门控和字段拆解，构建信息密集且可控的长上下文输入。

8.3 对“多轮交互一致性与可扩展性”的响应
通过两阶段流程、复核即时回写、可指定起始题号复核等机制，实现可持续迭代与大规模执行。


九、结论

本方案围绕赛题三大核心问题，构建了可复现、可扩展、可持续优化的超长上下文数据标注系统。其核心价值在于：
1. 以结构化提示工程提升任务稳定性。
2. 以超长样本筛选与信息重排提升上下文利用效率。
3. 以两阶段复核机制提升结果与工程可靠性。

在当前代码基础上，该方案已形成端到端可运行闭环，可直接用于竞赛阶段结果提交与技术评审复现。
